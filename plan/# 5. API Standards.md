# 5. API Standards

API Standards define **how every microservice exposes APIs**. Consistent standards make APIs easier to understand, maintain, secure, and document.

---

# 5.1 API Design Principles

### Use RESTful APIs

* Resource-oriented URLs
* Standard HTTP methods
* Stateless communication
* JSON request/response bodies

Example:

```http
GET /api/v1/products
```

---

### Version APIs

Always include the version in the URL.

```text
/api/v1/...
```

Examples

```http
GET    /api/v1/products
POST   /api/v1/orders
PUT    /api/v1/users/me
DELETE /api/v1/cart/items/{itemId}
```

---

### Use Nouns, Not Verbs

✅ Good

```text
/products
/orders
/users
/categories
/cart
/payments
```

❌ Bad

```text
/getProducts
/createProduct
/updateUser
/deleteOrder
```

---

# 5.2 HTTP Methods

| Method | Purpose          | Idempotent |
| ------ | ---------------- | ---------- |
| GET    | Read data        | ✅          |
| POST   | Create resource  | ❌          |
| PUT    | Replace resource | ✅          |
| PATCH  | Partial update   | ❌          |
| DELETE | Delete resource  | ✅          |

Example

```http
GET /products
```

```http
POST /orders
```

```http
PUT /products/{id}
```

```http
PATCH /users/me
```

```http
DELETE /cart/items/{itemId}
```

---

# 5.3 URL Naming Convention

Use:

* lowercase
* plural nouns
* hyphens for multi-word resources

Example

```text
/api/v1/products

/api/v1/product-categories

/api/v1/order-items
```

Avoid

```text
/api/Product

/api/getProduct

/api/order_item
```

---

# 5.4 Request Headers

Standard headers:

```http
Content-Type: application/json
Accept: application/json
Authorization: Bearer <JWT>
X-Request-Id: <UUID>
X-Correlation-Id: <UUID>
```

### Why?

* `Authorization` → user authentication.
* `X-Request-Id` → trace an individual request.
* `X-Correlation-Id` → trace a request across multiple microservices.

---

# 5.5 Standard Request Body

Example

```json
{
  "name": "Gaming Laptop",
  "price": 85000,
  "categoryId": "CAT001"
}
```

Rules

* camelCase field names
* Validate all inputs
* Ignore unknown fields where appropriate or reject them consistently
* Never trust client input

---

# 5.6 Standard Response Format

Every API should return a consistent response structure.

### Success

```json
{
  "success": true,
  "message": "Product created successfully.",
  "data": {
    "id": "P123",
    "name": "Gaming Laptop"
  },
  "meta": {
    "timestamp": "2026-07-19T10:30:00Z",
    "requestId": "7d1e5d4b-9e8d-4f2f-8c1a-5d1f0c4d3e2b"
  }
}
```

---

### Error

```json
{
  "success": false,
  "message": "Validation failed.",
  "errors": [
    {
      "field": "price",
      "message": "Price must be greater than zero."
    }
  ],
  "meta": {
    "timestamp": "2026-07-19T10:30:00Z",
    "requestId": "7d1e5d4b-9e8d-4f2f-8c1a-5d1f0c4d3e2b"
  }
}
```

# 5.7 HTTP Status Codes

| Status | Meaning               | Usage                                   |
| ------ | --------------------- | --------------------------------------- |
| 200    | OK                    | Successful GET/PUT/PATCH                |
| 201    | Created               | Resource created                        |
| 202    | Accepted              | Asynchronous processing started         |
| 204    | No Content            | Successful delete                       |
| 400    | Bad Request           | Invalid input                           |
| 401    | Unauthorized          | Missing or invalid authentication       |
| 403    | Forbidden             | Authenticated but not authorized        |
| 404    | Not Found             | Resource doesn't exist                  |
| 409    | Conflict              | Duplicate resource or business conflict |
| 422    | Unprocessable Entity  | Business validation failed              |
| 429    | Too Many Requests     | Rate limit exceeded                     |
| 500    | Internal Server Error | Unexpected server error                 |

# 5.8 Pagination

Request

```http
GET /api/v1/products?page=1&pageSize=20
```

Response

```json
{
  "success": true,
  "data": [...],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "totalItems": 540,
    "totalPages": 27
  }
}
```

---

# 5.9 Filtering

```http
GET /api/v1/products?category=electronics
```

```http
GET /api/v1/products?brand=apple
```

```http
GET /api/v1/products?minPrice=1000&maxPrice=5000
```

```http
GET /api/v1/products?rating=4
```

Filters can be combined:

```http
GET /api/v1/products?category=electronics&brand=apple&minPrice=50000&maxPrice=100000
```

# 5.10 Sorting

```http
GET /api/v1/products?sort=price
```

Descending

```http
GET /api/v1/products?sort=-price
```

Multiple fields

```http
GET /api/v1/products?sort=-rating,price
```

# 5.12 Idempotency

For operations like payment creation, retries should not create duplicates.

Clients send:

```http
Idempotency-Key: 3b5d4c0d-0d78-4e98-8f4e-5ef9f5bde123
```

If the same key is sent again, the server returns the original result instead of creating a new payment or order.

Use this for:

* Payment creation
* Order creation
* Refund processing

# 5.13 Authentication & Authorization

Protected endpoints require:

```http
Authorization: Bearer <JWT>
```

Example

```http
GET /api/v1/orders
```

JWT contains:

```json
{
  "sub": "USER123",
  "roles": ["Customer"],
  "permissions": ["order.read", "order.create"]
}
```

Authorization should be permission-based (RBAC).

---

# 5.14 Validation

Validate:

* Required fields
* String length
* Email format
* Phone format
* Enum values
* Numeric ranges
* UUID/ID format

Example

```json
{
  "email": "invalid-email"
}
```

Response

```json
{
  "success": false,
  "message": "Validation failed.",
  "errors": [
    {
      "field": "email",
      "message": "Invalid email format."
    }
  ]
}
```

---

# 5.15 API Documentation

Every API should be documented using **OpenAPI (Swagger)**.

Each endpoint should include:

* Description
* Request parameters
* Request body schema
* Response schema
* Authentication requirements
* Error responses
* Example requests and responses

# 5.16 Naming Conventions

| Item                | Convention              | Example               |
| ------------------- | ----------------------- | --------------------- |
| URL                 | kebab-case              | `/product-categories` |
| JSON fields         | camelCase               | `productName`         |
| Enum values         | UPPER_SNAKE_CASE        | `PAYMENT_SUCCESS`     |
| Event names         | PascalCase (past tense) | `OrderCreated`        |
| Database tables     | snake_case              | `order_items`         |
| Columns             | snake_case              | `created_at`          |
| TypeScript classes  | PascalCase              | `OrderService`        |
| Variables/functions | camelCase               | `createOrder()`       |

---

# 5.17 Standard API Flow

```text
Client
   │
   ▼
API Gateway
   │
   ├── Authenticate JWT
   ├── Rate Limit
   ├── Validate Request
   └── Route Request
   │
   ▼
Microservice
   │
   ├── Validate Business Rules
   ├── Execute Use Case
   ├── Save to Database
   ├── Publish Domain Event (if applicable)
   └── Return Standard Response
```

# API Standards Checklist

| Standard                            | Status |
| ----------------------------------- | ------ |
| RESTful endpoints                   | ✅      |
| Versioned APIs (`/api/v1`)          | ✅      |
| JSON request/response               | ✅      |
| Consistent response format          | ✅      |
| Standard HTTP status codes          | ✅      |
| Pagination, filtering, sorting      | ✅      |
| JWT authentication                  | ✅      |
| RBAC authorization                  | ✅      |
| Input validation                    | ✅      |
| Idempotency for critical operations | ✅      |
| OpenAPI/Swagger documentation       | ✅      |
| Request and correlation IDs         | ✅      |

These standards should be applied consistently across all microservices so that every API behaves predictably, regardless of which team or service implements it.
