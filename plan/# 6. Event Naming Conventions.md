# 6. Event Naming Conventions

In a microservices architecture, **events** are used to notify other services that something important has already happened. Good event naming makes the system easier to understand, maintain, and extend.

---

# 6.1 Event Naming Principles

### 1. Use Past Tense

An event represents something that **already happened**.

✅ Good

```text
OrderCreated
PaymentSucceeded
InventoryReserved
ShipmentDelivered
```

❌ Bad

```text
CreateOrder
PayNow
ReserveInventory
DeliverShipment
```

Commands (requests) use verbs, while **events use past-tense verbs**.

---

### 2. Use PascalCase

Event names should use PascalCase.

✅ Good

```text
UserRegistered
OrderCancelled
PaymentFailed
```

❌ Bad

```text
user_registered
order-created
payment_failed
```

---

### 3. Be Business-Oriented

Events should describe **business facts**, not technical actions.

✅ Good

```text
OrderCreated
RefundCompleted
ProductReviewed
```

❌ Bad

```text
DatabaseUpdated
RowInserted
CacheInvalidated
```

---

### 4. One Event = One Business Action

Each event should represent one completed business action.

Example

```text
OrderCreated
```

Not

```text
OrderCreatedAndPaid
```

Instead publish two events:

```text
OrderCreated

PaymentSucceeded
```

# 6.3 Event Payload Standard

Every event should follow a common structure.

```json
{
  "eventId": "evt-123456",
  "eventType": "OrderCreated",
  "eventVersion": "1.0",
  "occurredAt": "2026-07-19T10:30:00Z",
  "source": "order-service",
  "correlationId": "req-123456",
  "data": {
    "orderId": "ORD001",
    "customerId": "CUS001",
    "totalAmount": 2500
  }
}
```

---

# 6.4 Required Event Metadata

| Field         | Description                   |
| ------------- | ----------------------------- |
| eventId       | Unique event identifier       |
| eventType     | Event name                    |
| eventVersion  | Schema version                |
| occurredAt    | Event timestamp (UTC)         |
| source        | Publishing service            |
| correlationId | Trace request across services |
| data          | Business payload              |

Optional fields:

```text
tenantId
userId
traceId
causationId
metadata
```

---

# 6.5 Event Versioning

Never modify an existing event schema in a breaking way.

Instead:

```text
OrderCreated v1

↓

OrderCreated v2
```

or

```text
eventVersion = "2.0"
```

Consumers can migrate gradually.

# 6.7 Event Flow Example

Customer places an order.

```text
Customer
    │
    ▼
Order Service
    │
    ├── Publish → OrderCreated
    │
RabbitMQ / Kafka
    │
    ├────────► Inventory Service
    │              │
    │              └── Publish → InventoryReserved
    │
    ├────────► Payment Service
    │              │
    │              └── Publish → PaymentSucceeded
    │
    ├────────► Notification Service
    │              │
    │              └── Send Confirmation
    │
    └────────► Analytics Service
```

# 6.8 Event Topics (Kafka/RabbitMQ)

A common naming convention is:

```text
auth.user.registered
user.profile.updated

product.created
product.updated
product.deleted

inventory.reserved
inventory.released

cart.checked-out

order.created
order.cancelled
order.completed

payment.succeeded
payment.failed

shipment.created
shipment.delivered

review.added

notification.sent

support.ticket.created
```

Use:

* lowercase
* dot (`.`) separator
* singular domain name
* past-tense action

# 6.9 Event Naming Checklist

| Rule                           | Example                                         |
| ------------------------------ | ----------------------------------------------- |
| Use past tense                 | `OrderCreated`                                  |
| Use PascalCase for event types | `PaymentSucceeded`                              |
| Keep names business-focused    | `RefundCompleted`                               |
| One event per business action  | `ShipmentDelivered`                             |
| Include standard metadata      | `eventId`, `occurredAt`, `source`, etc.         |
| Version schemas                | `eventVersion: "1.0"`                           |
| Use dot-separated topic names  | `order.created`                                 |
| Avoid technical event names    | Don't use `DatabaseUpdated` or `CacheRefreshed` |

---

## Event vs Command

A common source of confusion is the difference between **commands** and **events**:

| Command            | Event                                |
| ------------------ | ------------------------------------ |
| `CreateOrder`      | `OrderCreated`                       |
| `ReserveInventory` | `InventoryReserved`                  |
| `ProcessPayment`   | `PaymentSucceeded` / `PaymentFailed` |
| `ShipOrder`        | `ShipmentDispatched`                 |
| `SendNotification` | `NotificationSent`                   |

* **Commands** express an intention: "Please do this."
* **Events** announce a fact: "This has already happened."

Using this convention consistently makes event-driven workflows much easier to understand and maintain across all microservices.
