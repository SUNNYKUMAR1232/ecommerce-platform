# 7. Git Strategy

A Git strategy defines **how the team collaborates on the codebase**, manages releases, and maintains code quality.

For a production microservices project, use **GitHub Flow** (or a simplified Git Flow) with **Trunk-Based principles**:

* `main` is always deployable.
* Development happens in short-lived feature branches.
* Changes are merged through Pull Requests after review and CI checks.

---

# 7.1 Repository Strategy

There are two common approaches:

## Option 1: Monorepo (Recommended)

One repository contains all services.

```text
ecommerce-platform/
│
├── apps/
│   ├── api-gateway/
│   ├── auth-service/
│   ├── user-service/
│   ├── product-service/
│   ├── cart-service/
│   ├── inventory-service/
│   ├── order-service/
│   ├── payment-service/
│   ├── shipping-service/
│   ├── review-service/
│   ├── notification-service/
│   ├── search-service/
│   ├── recommendation-service/
│   ├── support-service/
│   ├── analytics-service/
│   └── admin-service/
│
├── packages/
│   ├── common/
│   ├── config/
│   ├── logger/
│   ├── events/
│   ├── auth/
│   └── types/
│
├── infrastructure/
│   ├── docker/
│   ├── kubernetes/
│   └── terraform/
│
└── docs/
```

### Advantages

* Easier code sharing
* One CI/CD pipeline
* Consistent dependency versions
* Simpler refactoring across services
* Better developer experience

---

## Option 2: Polyrepo

Each service has its own repository.

```text
auth-service
user-service
product-service
order-service
payment-service
...
```

### Advantages

* Complete service independence
* Independent release cycles
* Strong isolation

### Disadvantages

* Harder dependency management
* More CI/CD pipelines
* More repository administration

For your project, **Monorepo** is the better choice.

# 7.2 Branch Strategy

```text
main
│
├── feature/auth-login
├── feature/product-crud
├── feature/cart-service
├── feature/order-api
├── bugfix/payment-timeout
├── hotfix/security-patch
└── release/v1.0.0
```

---

## Main Branch

```text
main
```

Rules:

* Always deployable
* Protected branch
* No direct commits
* Merge only via Pull Request

---

## Feature Branches

Naming:

```text
feature/<feature-name>
```

Examples:

```text
feature/auth-service
feature/product-crud
feature/search-api
feature/payment-integration
```

Used for:

* New features
* Enhancements

---

## Bug Fix Branches

Naming:

```text
bugfix/<issue-name>
```

Examples:

```text
bugfix/cart-total
bugfix/payment-timeout
bugfix/search-filter
```

---

## Hotfix Branches

Naming:

```text
hotfix/<issue-name>
```

Examples:

```text
hotfix/jwt-expiry
hotfix/security-patch
```

Used for urgent production fixes.

---

## Release Branches (Optional)

Naming:

```text
release/v1.0.0
release/v1.1.0
```

Used to stabilize a version before production deployment.

# 7.4 Commit Message Convention

Follow the **Conventional Commits** specification.

```
<type>: <short description>
```

Common types:

| Type     | Purpose                                    |
| -------- | ------------------------------------------ |
| feat     | New feature                                |
| fix      | Bug fix                                    |
| docs     | Documentation                              |
| refactor | Code restructuring without behavior change |
| test     | Tests                                      |
| chore    | Maintenance tasks                          |
| perf     | Performance improvements                   |
| ci       | CI/CD changes                              |
| build    | Build system or dependencies               |
| style    | Formatting changes                         |

Examples:

```text
feat(auth): add OTP login

feat(product): implement product CRUD

fix(order): prevent duplicate order creation

refactor(cart): simplify price calculation

docs(api): add Swagger documentation

test(payment): add refund unit tests

ci(github): add deployment workflow
```

---

# 7.5 Pull Request Guidelines

Every Pull Request should include:

* Meaningful title
* Description of changes
* Linked issue (if applicable)
* Testing evidence
* Updated documentation (if required)

Checklist:

```text
☑ Code compiles
☑ Tests pass
☑ Lint passes
☑ No secrets committed
☑ API documentation updated
☑ Database migration reviewed
```

---

# 7.6 Branch Protection Rules

Protect the `main` branch with:

* No direct pushes
* Pull Request required
* At least one approval (or two for larger teams)
* All CI checks passing
* Resolved conversations before merge

---

# 7.7 Versioning

Use **Semantic Versioning (SemVer)**:

```
MAJOR.MINOR.PATCH
```

Examples:

```
1.0.0
1.1.0
1.1.1
2.0.0
```

| Increment | Meaning                       |
| --------- | ----------------------------- |
| MAJOR     | Breaking API changes          |
| MINOR     | Backward-compatible features  |
| PATCH     | Backward-compatible bug fixes |

---

# 7.8 Tags

Create a Git tag for each release.

```text
v1.0.0

v1.1.0

v2.0.0
```

# 7.9 GitHub Actions Workflow

Each Pull Request should trigger:

```text
Pull Request
     │
     ▼
Install Dependencies
     │
     ▼
Lint
     │
     ▼
Unit Tests
     │
     ▼
Integration Tests
     │
     ▼
Build
     │
     ▼
Security Scan
     │
     ▼
Merge
```

After merging to `main`:

```text
Merge
   │
Build Docker Image
   │
Push Image
   │
Deploy to Kubernetes
```

---

# 7.10 Ignore Files

Example `.gitignore`:

```text
node_modules/
dist/
coverage/
.env
.env.*
*.log
.vscode/
.idea/
```

**Never commit:**

* API keys
* JWT secrets
* Database passwords
* Cloud credentials
* SSL certificates
* Production `.env` files

Use a secrets manager or CI/CD secret store instead.

# 7.11 Code Review Guidelines

Review for:

* Correctness

* Performance

# 7.12 Git Strategy Summary

| Area              | Standard                    |
| ----------------- | --------------------------- |
| Repository        | Monorepo                    |
| Default branch    | `main`                      |
| Feature branches  | `feature/<name>`            |
| Bug fixes         | `bugfix/<name>`             |
| Hotfixes          | `hotfix/<name>`             |
| Releases          | `release/vX.Y.Z` (optional) |
| Commit format     | Conventional Commits        |
| Pull Requests     | Required for `main`         |
| Branch protection | Enabled                     |
| Versioning        | Semantic Versioning         |
| Tags              | `vX.Y.Z`                    |
| CI/CD             | GitHub Actions              |
| Deployment        | Automated from `main`       |

---

## Recommended Workflow for Your Project

Since you're building a **NestJS + Next.js microservices** platform, the recommended approach is:

* **Monorepo** managed with **Nx** or **Turborepo**.
* **GitHub Flow** with short-lived feature branches.
* **Conventional Commits** for clear history and automated release notes.
* **Protected `main` branch** with mandatory Pull Requests and passing CI.
* **Automated Docker builds and Kubernetes deployments** from `main`.

This approach scales well from a single developer to a large engineering team while keeping the development process simple and maintainable.
