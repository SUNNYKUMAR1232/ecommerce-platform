# 8. Coding Standards

Coding standards define the rules and best practices that every developer should follow to ensure the codebase is **consistent, readable, maintainable, secure, and scalable**.

---

# 8.1 General Principles

Follow these software engineering principles:

* Write clean and readable code.
* Keep solutions simple (KISS).
* Don't Repeat Yourself (DRY).
* Don't build unnecessary features (YAGNI).
* Follow the Single Responsibility Principle (SRP).
* Prefer composition over inheritance.
* Write self-documenting code.
* Keep business logic separate from infrastructure concerns.

---

# 8.2 Project Structure

Organize the project logically.

```text
Project
│
├── Source Code
├── Configuration
├── Documentation
├── Tests
├── Scripts
├── Assets
└── Build Output
```

Each module should have a clear responsibility.

---

# 8.3 Naming Conventions

Use meaningful and consistent names.

### Classes

```text
Order
Product
Customer
```

### Functions

```text
createOrder()
calculateTotal()
sendNotification()
```

### Variables

```text
orderTotal
customerName
productPrice
```

### Constants

```text
MAX_LOGIN_ATTEMPTS
DEFAULT_PAGE_SIZE
```

### Files

Use lowercase with separators consistently.

Example

```text
order-service
payment-handler
user-profile
```

---

# 8.4 Code Organization

* One responsibility per module.
* One responsibility per class.
* Functions should perform one task.
* Keep files focused on a single feature or concern.
* Avoid very large files and classes.

---

# 8.5 Functions

Functions should:

* Be small.
* Have descriptive names.
* Accept only necessary parameters.
* Return predictable results.
* Avoid unnecessary side effects.

Good

```text
calculateTax()
```

Bad

```text
calculateTaxAndSaveOrderAndSendEmail()
```

---

# 8.6 Error Handling

* Handle errors gracefully.
* Use meaningful error messages.
* Don't expose internal implementation details.
* Log errors appropriately.
* Validate all external input.

---

# 8.7 Input Validation

Always validate:

* Required fields
* Data types
* Length limits
* Numeric ranges
* Date formats
* Email formats
* Phone numbers
* Business rules

Never trust user input.

---

# 8.8 Logging

Log important events such as:

* User actions
* System events
* Errors
* Warnings
* Performance information

Do **not** log:

* Passwords
* Tokens
* Payment information
* Personal sensitive information
* Secret keys

---

# 8.9 Security

* Authenticate users before accessing protected resources.
* Authorize users based on permissions or roles.
* Encrypt sensitive data.
* Hash passwords.
* Sanitize user input.
* Protect against common security vulnerabilities.
* Store secrets securely.
* Use secure communication channels.

---

# 8.10 Database Practices

* Keep data consistent.
* Use transactions where needed.
* Avoid duplicate data.
* Index frequently queried fields.
* Use migrations for schema changes.
* Never access another service's database directly in a microservices architecture.

---

# 8.11 API Design

* Use consistent naming.
* Version APIs.
* Return consistent response formats.
* Use standard HTTP methods and status codes.
* Support pagination for large datasets.
* Validate all requests.

---

# 8.12 Event Standards

For event-driven systems:

* Use meaningful event names.
* Use past-tense event names.
* Include timestamps.
* Include unique event identifiers.
* Version event schemas when changes are required.

---

# 8.13 Documentation

Every module should include:

* Purpose
* Public interfaces
* Configuration requirements
* Usage examples
* Known limitations (if any)

Keep documentation updated with code changes.

---

# 8.14 Testing

Test different levels of the application:

* Unit tests
* Integration tests
* End-to-end tests

Test:

* Normal scenarios
* Edge cases
* Invalid inputs
* Error handling

---

# 8.15 Performance

* Avoid unnecessary computations.
* Optimize frequently executed code.
* Cache data when appropriate.
* Reduce network calls.
* Optimize database queries.
* Load resources efficiently.

---

# 8.16 Version Control

* Use meaningful commit messages.
* Keep commits small and focused.
* Review code before merging.
* Do not commit generated files, secrets, or credentials.

---

# 8.17 Code Review Checklist

Before merging code, verify:

* Code follows naming conventions.
* No duplicated logic.
* Functions and classes have a single responsibility.
* Error handling is implemented.
* Inputs are validated.
* Security considerations are addressed.
* Performance impacts are acceptable.
* Tests pass.
* Documentation is updated.

---
