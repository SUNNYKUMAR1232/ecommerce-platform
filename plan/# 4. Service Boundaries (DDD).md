# 4. Service Boundaries (DDD)

## 4.1 Overview

A **Service Boundary** defines the responsibility and ownership of a microservice.

Each service should have:

* A single business responsibility
* Its own database
* Its own APIs
* Its own domain logic
* Published and consumed events
* No direct access to another service's database

---

## 4.2 Service Boundary Principles

| Principle              | Description                                        |
| ---------------------- | -------------------------------------------------- |
| Single Responsibility  | One business capability per service                |
| High Cohesion          | Related business logic stays together              |
| Loose Coupling         | Services communicate through APIs or events        |
| Database per Service   | Each service owns its database                     |
| Independent Deployment | Services can be deployed independently             |
| Bounded Context        | Each service has its own domain model and language |

# 4.4 Service Interaction Overview

```text
Client
   │
API Gateway
   │
   ├── Auth Service
   ├── User Service
   ├── Product Service
   ├── Cart Service
   └── Order Service
            │
            ├──────── REST/gRPC ────────► Inventory Service
            ├──────── REST/gRPC ────────► Payment Service
            └──────── Publish Event ───► RabbitMQ/Kafka
                                           │
                 ┌─────────────────────────┼─────────────────────────┐
                 ▼                         ▼                         ▼
          Notification              Analytics                Shipping
                 │                         │                         │
                 └──────────────► Other interested consumers ◄──────┘
```

# 4.5 Boundary Rules

1. Each service owns its **business logic** and **database**.
2. No service accesses another service's database directly.
3. Use **REST/gRPC** for synchronous request-response operations.
4. Use **RabbitMQ/Kafka** for asynchronous domain events.
5. Share only **IDs** (for example, `productId`, `userId`, `orderId`) across service boundaries rather than entire objects.
6. Each service can be developed, tested, deployed, and scaled independently.

This document serves as the contract for your microservices architecture and becomes the foundation for the Low-Level Design (LLD) of each service.
