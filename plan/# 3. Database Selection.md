# 3. Database Selection

Database selection is one of the most important design decisions in a microservices architecture. The principle is:

> **Each microservice owns its own database.** No other service can read or write that database directly. Communication happens through APIs or events.

This is called **Database per Service** and helps keep services independent.

# 1. Auth Service

### Database

**PostgreSQL**

### Why?

Authentication data requires:

* ACID transactions
* Strong consistency
* Foreign keys
* Secure storage

### Stores

```text
Users

Roles

Permissions

Refresh Tokens

Sessions

Devices
```

---

# 2. User Service

### Database

**PostgreSQL**

### Why?

User data is relational.

Example

```text
User
 │
 ├── Address
 ├── Email
 ├── Phone
 └── Preferences
```

PostgreSQL handles these relationships well.

---

# 3. Product Service

### Database

**MongoDB**

### Why?

Products have different attributes.

Example:

Phone

```json
{
  "name":"iPhone",
  "ram":"8GB",
  "storage":"256GB"
}
```

Shoes

```json
{
  "name":"Nike",
  "size":[7,8,9],
  "color":["Black","White"]
}
```

Laptop

```json
{
  "cpu":"Intel",
  "gpu":"RTX 5070",
  "ram":"32GB"
}
```

A relational schema would require many optional columns or additional tables. MongoDB's flexible document model handles these variations naturally.

---

# 4. Cart Service

### Database

**Redis**

### Why?

Shopping carts:

* Change frequently
* Need very fast access
* Are temporary
* Can expire automatically

Redis stores data in memory and supports TTL (Time To Live).

Example

```text
Cart

↓

Expire after 30 days
```

---

# 5. Inventory Service

### Database

**PostgreSQL**

### Why?

Inventory requires strict consistency.

Example

```text
Stock = 5

Customer A buys 3

Customer B buys 3
```

Without transactions you could oversell.

PostgreSQL prevents this using row locking and transactions.

---

# 6. Order Service

### Database

**PostgreSQL**

### Why?

Orders require:

* ACID transactions
* Consistency
* Referential integrity
* Reliable updates

Example

```text
Order

↓

Order Items

↓

Invoice
```

---

# 7. Payment Service

### Database

**PostgreSQL**

### Why?

Financial systems require:

* ACID
* Rollback
* Audit
* Reliability

Never use a NoSQL database as the primary store for payment records unless you have a specific design that guarantees financial correctness.

---

# 8. Shipping Service

### Database

**PostgreSQL**

### Why?

Shipment data is structured.

Example

```text
Shipment

↓

Courier

↓

Tracking

↓

Delivery Status
```

---

# 9. Review Service

### Database

**MongoDB**

### Why?

Reviews contain flexible content.

Example

```json
{
 "rating":5,
 "review":"Amazing",
 "images":[...],
 "videos":[...]
}
```

Some reviews have media, others don't. MongoDB adapts easily.

---

# 10. Notification Service

### Database

**MongoDB**

### Why?

Notifications have different payloads.

Email

```json
{
 "subject":"Order Confirmed"
}
```

SMS

```json
{
 "otp":"123456"
}
```

Push

```json
{
 "title":"Sale"
}
```

A document database is a good fit for these varying structures.

---

# 11. Search Service

### Database

**Elasticsearch / OpenSearch**

### Why?

Designed for:

* Full-text search
* Filtering
* Auto-complete
* Ranking
* Faceted search

Example

Search

```text
Gaming Laptop
```

Returns

* Matching products
* Suggestions
* Highlighted results

This is much faster and more feature-rich than SQL `LIKE` queries.

---

# 12. Recommendation Service

### Database

**Neo4j**

### Why?

Recommendations depend on relationships.

Example

```text
Customer

↓

Bought

↓

Laptop

↓

Bought

↓

Mouse

↓

Bought

↓

Keyboard
```

Graph databases traverse these relationships efficiently.

---

# 13. Analytics Service

### Database

**ClickHouse**

### Why?

Analytics involves scanning large volumes of data.

Questions like:

```text
Total Sales

↓

Monthly Revenue

↓

Top Products

↓

Best Cities
```

ClickHouse is optimized for fast analytical queries over millions of records.

---

# Database Selection Principles

When choosing a database, ask these questions:

| Question                                   | Best Choice              |
| ------------------------------------------ | ------------------------ |
| Do I need ACID transactions?               | PostgreSQL               |
| Is the schema highly variable?             | MongoDB                  |
| Is extremely fast in-memory access needed? | Redis                    |
| Do I need full-text search?                | Elasticsearch/OpenSearch |
| Is the data primarily relationships?       | Neo4j                    |
| Will I run large analytical queries?       | ClickHouse               |

---

## Final Database Architecture

```text
                    E-Commerce Platform

 Auth ---------------------- PostgreSQL

 User ---------------------- PostgreSQL

 Product ------------------- MongoDB

 Cart ---------------------- Redis

 Inventory ----------------- PostgreSQL

 Order --------------------- PostgreSQL

 Payment ------------------- PostgreSQL

 Shipping ------------------ PostgreSQL

 Review -------------------- MongoDB

 Notification -------------- MongoDB

 Search -------------------- Elasticsearch/OpenSearch

 Recommendation ------------ Neo4j

 Analytics ----------------- ClickHouse
```

This approach is called **Polyglot Persistence**: each microservice uses the database technology that best matches its workload instead of forcing one database to solve every problem.
